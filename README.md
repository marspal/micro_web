# micro_web

## 软件设计分层

> 架构种类

1. 系统级架构
   应用在整个系统内, 如与后台服务如何通信、与第三方系统如何集成；

   前提: 了解前端系统与其他系统之间的关系
   关系包括: 业务关系和协作机制

   设计后端: 只需要规定与后端数据传递机制
   包括: api 设计规则,访问授权(OAuth)token 跳转验证、数据传递 Cookie 等

   > 微前端: 将多个应用以某种方式组合在一起

在一个系统内微前端是应用间的架构方案
在多个应用之间，微前端是一种系统间等架构方案

单实例: 即同一时刻、只有一个子应用被展示，子应用具备一个完整的应用生命周期
多实例: 通常基于 url 变化来做子应用的切换; 同一时刻课展示多个子应用; 通常使用 Web Components 方案来做子应用封装,子应用更像一个业务组件

2. 应用级架构
   系统级架构的实现，含脚手架、模式库、设计系统
3. 模块级架构
   编码前进行设计，我们称为迭代
4. 代码级架构
   规范与原则

> 系统的质量

1. 健壮性
   计算机软件在输入错误、磁盘故障、网络过载或者有意攻击的情况下、能否不死机 不崩溃， 就是具体
   简而言之: 一个系统容错能力强、运行不易被干扰、安全性好

> 架构师分类

1. 系统架构师

2. 应用架构师

3. 业务架构师

> 技术前期准备

1. 技术选型: 社区氛围、发展规模、未来发展趋势；与当前团队的切合程度、执行成本、维护和迁移成本、执行效率等内容的调研和报告
2. 充分调研每一项技术的利弊
3. 最大程度上预测架构设计中的缺陷、以防止问题的发生
